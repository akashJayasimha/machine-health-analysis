Step 02
Sensor trends — finding the signal
Raw sensor data is noisy. The analyst's job is to identify which signals carry predictive weight and which are just variance. Two sensors stood out: temperature drift and vibration levels. The key question — are these independent, or do they move together?

Step 03
Failure patterns & downtime cost
Failure frequency and downtime are different problems. A machine that fails ten times but recovers in 30 minutes is less costly than one that fails twice but takes 12 hours to repair each time. This step separates count from impact.
Failure events vs. estimated downtime — by machine
bubble size = age
The top-right quadrant — high failures, high downtime — is where MCH-004 sits alone. Age amplifies both frequency and recovery time, creating a compounding cost.
Observation 3.1
MCH-004 has 2 failure events in 90 days — nearly double any other asset. At an estimated 5 hours of downtime, this single machine likely represents the majority of unplanned maintenance cost. This is where to act first.
Step 04
A simple risk classifier
Machine learning is not always the answer. Before reaching for complex models, a well-designed threshold-based classifier can be interpretable, auditable, and immediately actionable — especially with limited labelled data. The goal here is clarity, not sophistication.
Python · threshold classifier
def classify(row):
    score = 0
    if row['avg_temp']  > 85: score += 2
    elif row['avg_temp']> 75: score += 1
    if row['avg_vib']   > 4:  score += 2
    elif row['avg_vib'] > 2.5:score += 1
    if row['age']       > 8:  score += 2
    elif row['age']     > 5:  score += 1
    if row['failures']  > 5:  score += 1
    if   score >= 5: return 'Critical'
    elif score >= 3: return 'At Risk'
    else:            return 'Stable'

summary['risk_class'] = summary.apply(classify, axis=1)

Model Note
The classifier is deliberately transparent — each rule maps to a physical reality that a maintenance engineer can validate. An unexplainable model that produces the right answer is less useful than an explainable one that can be trusted and refined over time.
Step 05
Recommendations & next steps
Analysis only has value if it leads to action. The following recommendations are ranked by expected impact — not by analytical complexity.
01
Schedule immediate inspection of MCH-004
Temperature is trending toward the 90°C operating limit with no sign of stabilising. Bearing wear is the likely root cause given the vibration profile and asset age.
Impact: High · Timeline: Immediate
02
Establish vibration alert thresholds in the monitoring system
The data shows that vibration above 3 mm/s reliably precedes temperature spikes by 4–7 days. This lag is a usable early warning window — it should be automated, not caught manually.
Impact: High · Timeline: 2 weeks
03
Review asset retirement policy for machines older than 8 years
Age is the strongest predictor of degradation in this dataset. A cost-benefit model comparing continued maintenance vs. replacement for MCH-004 and MCH-002 is warranted.
Impact: Medium · Timeline: Next planning cycle
04
Expand sensor coverage and begin labelling failure types
The current dataset conflates all failure events into a binary flag. Distinguishing mechanical from electrical failures would unlock a more precise predictive model in the next iteration.
Impact: Medium (future) · Timeline: Q2 2025
